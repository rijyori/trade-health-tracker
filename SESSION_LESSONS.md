# Deepcoin 프로젝트 — 세션 결정사항/디버깅 다이제스트

`ARCHITECTURE.md`가 "구조"를 정리한 문서라면, 이 문서는 **실제로 겪었던 구체적 사건과 수치**를

## 1. 레이트리밋은 전역 1/s가 아니라 엔드포인트별로 다르다

처음엔 "1 req/sec 전역"이라 가정하고 모든 API 호출에 1.1초 스로틀을 걸었음. 나중에 실측(동시
요청 30개를 쏴서 429 받을 때까지 밀어붙이고 응답 헤더 확인)해보니:

```
/trade/orders-history : 5/s
/trade/fills          : 5/s
/account/bills         : 5/s
/account/balances      : 10/s
/account/positions     : 10/s
/market/instruments    : 10/s
```

응답 헤더에 `x-ratelimit-limit`, `x-ratelimit-window`가 그대로 찍힘 (429 status는 401로
오고 `code:"50000", msg:"Trigger the api frequency limiting"`). 엔드포인트별로 리밋이
다르고 다 1보다 훨씬 크다는 걸 확인 후, 스로틀 키를 `apiKey::path`로 바꿔서 5~10배 빨라짐.
**교훈: 문서/가정 믿지 말고 실측할 것.**

## 2. 커서 페이지네이션이 무한루프/순환할 수 있다

`trade/fills`를 페이지네이션할 때, 실제 데이터가 소진된 후에도 API가 같은 커서를 반복 반환해서
무한히 "새 데이터"인 것처럼 계속 요청하는 버그가 있었음. 마지막 레코드의 커서가 직전 페이지와
같으면 중단하는 로직(`fetchAllPages`의 stuck 감지)으로 해결.

## 3. [정정됨 — §14 참조] `orders-history`가 최근 구간만 주는 건 리텐션이 아니라 `before` 커서 버그였다

원래 여기 "리텐션 문제, 매일 refresh가 유일한 축적 수단"이라고 적혀 있었는데 **틀렸음.** 이
프로젝트에서 Deepcoin을 붙이다가 정확히 이 오판을 한 번 더 반복했고(§14), 최종적으로
`before` 커서가 이 엔드포인트에서 사실상 무시된다는 게 원인으로 확정됐다. 자세한 경위와 확정
실측 결과는 아래 **§14**를 볼 것 — 결론만: `cursorParam`을 `'after'`로 바꾸면 6개월치가 한
번의 refresh로 다 들어온다.

## 4. pnl은 GROSS다 (수수료 미포함) — 실측으로 검증

사용자가 "pnl에 이미 수수료가 포함된 거 아니냐"고 의심 → 실제 체결가로 (청산가-진입가)×수량×
계약승수를 역산해서 `pnl` 필드와 비교 → 일치 확인. **fee는 완전히 별도 필드, Net PNL = pnl -
ABS(fee) + rebate를 항상 명시적으로 계산해야 함.**

## 5. 리베이트 정산 규칙: UTC+8 경계, T+1, 정확히 32%

사용자가 딥코인 실제 커미션 지갑 잔고($18)와 우리 추정치가 안 맞다고 문제 제기. 디버깅 과정:

1. `/deepcoin/agents/users/rebates` 호출 → `totalCommission` 필드. 처음엔 이게 "다운라인
   레퍼럴 커미션"인 줄 알았는데, 사용자가 "그 uid 내 거임"이라고 정정 → 본인 계정 리베이트였음.
2. `/deepcoin/agents/users` → `rebateRate: "0.32"` 확인. 32%가 실측치임을 확인.
3. 사용자가 커미션 지갑의 "일별 입금 내역"(Time/Type/Size 표)을 수동으로 복사해서 줌.
4. 우리 쪽 일별 수수료 합계를 UTC+7/+8/+9로 각각 bucketing하고, lag 0/1일을 조합해서
   `commission[date] / fee[date-lag]` 비율이 가장 안정적으로 0.32에 수렴하는 조합을 탐색.
5. **UTC+8 경계 + T+1 정산**이 거의 모든 날짜에서 정확히 0.320으로 맞아떨어짐(소수점까지 일치).
6. 남은 오차(특정 날짜의 초과분)는 나중에 **서브계정을 대시보드에 합친 후** 그 서브계정의
   수수료를 더해보니 마지막 남은 미스매치까지 전부 해소됨 (메인+서브 fee 합 × 0.32 = 실제
   입금액, 22/24일 오차 0.0000).

**교훈: 정산 규칙(비율/시차/타임존)은 절대 문서로 안 나와 있고, 실측 데이터 매칭으로 역공학해야
알 수 있다.** 이 방법론(오프셋×lag 그리드 서치)은 다른 거래소에도 그대로 적용 가능.

## 6. 트레이드 그룹핑 버그: "아직 안 닫힌 포지션"이 실현손익을 통째로 삼킴

승률을 주문 단위에서 트레이드(포지션 생애주기) 단위로 바꾸는 리팩터 중, `short` 포지션 그룹의
사이즈 합이 끝까지 가도 정확히 0으로 안 떨어지고 1계약이 남는 케이스 발생. 처음엔 "아직 열려있는
포지션이라 그런가보다" 하고 넘어갈 뻔했는데, 사용자가 "나 지금 포지션 없는데?"라고 지적.

원인: 우리 주문내역 DB에 (리텐션 조회 윈도우 때문에) 과거 주문 일부가 빠져있어서, 사이즈
장부가 영구적으로 1만큼 안 맞았음(실제 포지션은 이미 flat). 이 상태에서 알고리즘이 "openQty가
정확히 0이 될 때만 트레이드를 확정 저장"하도록 짜여 있었기 때문에, **그 그룹의 마지막 27건
청산(이미 실현된 $15.17)이 통째로 통계에서 누락**되는 심각한 버그였음.

수정 두 단계:
1. 그룹 끝까지 갔는데 트레이드가 안 닫혀도 **일단 flush**(드롭하지 않음) — 이미 실현된 부분은
   보존.
2. 그 "open" 여부를 우리 자체 사이즈 계산으로 판단하지 말고, **라이브 `positions` 스냅샷과
   대조**해서 진짜 열려있을 때만 open=true로 표시. (자체 장부가 데이터 갭 때문에 신뢰 못 할
   수 있으므로, 외부 진실源과 항상 크로스체크.)

**교훈: 파생 상태(누적 사이즈 장부)를 신뢰의 근거로 삼지 말고, 가능하면 항상 라이브 API의
현재 상태와 대조 검증할 것. 특히 리텐션 갭이 있는 데이터 위에서는 더욱.**

## 7. "승률 뻥튀기" 문제와 트레이드 그룹핑 도입 배경

사용자 질문: "주문 단위로 승률 계산하면 부분익절 20번으로 나눠서 승률 뻥튀기 가능한 거 아니냐?"
→ 맞음, 확인됨. 근데 실제로 고쳐보니 **승률이 오히려 올라감**(19.9% → 29.7%, 버그 수정 후).
이유: 분할익절만 있는 게 아니라 **분할손절**도 있었고, 기존 방식은 그것도 개별 loss로 카운트
해서 승률을 깎아먹고 있었음. 게다가 "본절 잔량 정리"(수수료 때문에 순손실 나는 작은 마지막
청산)가 큰 익절 트레이드에 붙어서 개별 loss로 잘못 잡히던 것도 트레이드 단위로 묶이면서 정리됨.

**교훈: 메트릭을 order-level에서 trade-level로 바꿀 때 예상 방향(항상 낮아질 것)을 가정하지
말고 실제로 계산해서 확인할 것. 왜곡은 양방향으로 일어날 수 있다.**

## 8. IP 화이트리스트 API 키는 공유기/네트워크가 바뀌면 깨진다

메인 계정 API 키가 IP 화이트리스트로 바인딩돼 있었는데, 공유기 교체로 공인 IP가 바뀌면서
`401 Invalid IP (code 50110)` 발생. `https://api.ipify.org?format=json`로 현재 외부 IP를
확인하고, 거래소 대시보드에서 그 IP로 화이트리스트 갱신하면 해결. 서브계정 키는 IP 바인딩을
안 걸어놔서 영향 없었음.

## 9. 멀티 계정 병합: ID 유일성이 핵심 전제

주문 ID(`ordId`)/원장 ID(`billId`)가 시간순으로 엄격히 단조증가하는 걸 확인(계정이 달라도
겹치는 구간 없음) → 딥코인은 **계정을 구분하지 않는 전역 시퀀스**를 쓴다고 결론. 그래서
계정 태깅/복합키 없이 그냥 같은 테이블에 병합해도 안전했음.

신규 계정 추가 시 주의점: 공유 증분 커서(`begin=`)를 그대로 쓰면 신규 계정의 과거 데이터가
"이미 지나간 시점"으로 취급되어 전혀 안 긁힘 (다른 계정이 이미 최신까지 가 있으므로). 그래서
계정별로 `sync_state` 테이블에 "최초 동기화 여부"를 기록해서, 신규 계정은 첫 refresh만 무제한
조회(=조회 API가 주는 최대 범위)하고 이후부터 증분으로 전환하도록 함.

## 10. Node 동시성은 스레드 join과 다르다 (사용자 교육용으로 설명했던 내용)

`Promise.all`은 싱글 스레드 이벤트루프 위에서 I/O 대기를 겹치는 것뿐, 진짜 병렬 실행이 아님.
그래서 스레드 join에서 흔한 데드락/레이스컨디션 문제가 구조적으로 없음. 콜백은 항상 한 번에
하나씩만 실행됨. `Promise.all`은 "각 프로미스가 끝나길 기다리는" 것만 하므로, 개별 결과에
대한 후처리가 비동기라면 그 후처리까지 `.then()`으로 체이닝해서 `Promise.all`에 넘겨야 함
(안 그러면 후처리가 끝나기 전에 `Promise.all`이 먼저 resolve됨).

## 11. [trade-health-tracker 자체 발견] Gate.io 주문 ID는 JS Number 정밀도를 넘는다

Gate.io `/futures/usdt/orders`의 `id` 필드는 18~19자리 정수(예: `370421069416186023`)인데,
JS `Number`의 안전정수 범위(2^53-1, 약 16자리)를 넘어선다. 표준 `JSON.parse`로 파싱하면
**끝자리가 조용히 손상**된다(실측: `370421069416186023` → `370421069416186050`으로 변형,
에러 없이 발생). 이 값을 그대로 SQLite PK로 쓰면 주문이 뒤섞이거나 트레이드 그룹핑(FIFO)이
조용히 깨지는 심각한 데이터 손상으로 이어진다.

같은 응답 안에서도 `my_trades.order_id`나 `account_book.trade_id`는 API가 애초에 문자열로
반환해서 안전하지만, `orders.id`만 순수 숫자 리터럴로 옴 — 필드마다 다를 수 있으니 방심 금지.

**해결**: `res.json()` 대신 raw text를 받아 16자리 이상 순수 정수 리터럴을 정규식으로 문자열로
감싼 뒤 `JSON.parse`(`safeJsonParse`, `src/exchanges/gateio/client.js`). Deepcoin(OKX 계열)은
ID를 원래 문자열로 반환해서 이 문제가 없었던 것으로 보임 — **거래소마다 큰 정수 필드의 JSON
타입(문자열 vs 숫자)을 반드시 실측 확인할 것.**

## 12. [trade-health-tracker] Gate.io 이식 시 실측으로 확정한 것들 (2026-07-10)

- **PNL은 GROSS 확인**: `account_book`에서 같은 체결(trade_id)에 대해 `type:"pnl"`과
  `type:"fee"`가 별도 원장 항목으로 기록됨 → `orders.pnl`엔 수수료 미포함.
- **엔트리/청산 구분**: `orders.is_reduce_only` 플래그로 명확(false=엔트리, true=청산).
  Deepcoin의 `pnl===0` 추론보다 신뢰도 높음.
- **posSide 개념 없음(single 포지션 모드)**: Gate.io는 계약당 순포지션 하나만 허용하는
  `mode:"single"`이 기본. long/short 구분은 `is_reduce_only` + `size` 부호로 역산해야 함
  (엔트리는 size 부호=포지션 방향, 청산은 반대). `src/exchanges/gateio/fetcher.js`의
  `derivePosSide` 참고.
- **주문에 절대 수수료 없음**: `orders`엔 `mkfr`/`tkfr`(요율)만 있고 fee 금액이 없음.
  실제 fee는 `my_trades.fee`에 있고, 부분체결이면 여러 my_trades row로 쪼개져 있어서
  `order_id`로 묶어 합산해야 주문 단위 fee가 나옴.
- **positions는 미보유 계약까지 전부 내려옴** (`size:0`인 row 다수). 필터링 안 하면
  SESSION_LESSONS §6과 같은 클래스의 "가짜 오픈 포지션" 버그 재현. 반드시 `size !== 0`만 저장.
- **⚠️ 정정(같은 세션 내): "리텐션 2주"는 틀렸음 — 진짜는 최소 180일.**
  처음엔 `/futures/usdt/orders`와 `/futures/usdt/my_trades`에 `from`/`to`를 넘겨서 과거를
  긁으려 했는데, **이 두 엔드포인트는 애초에 `from`/`to` 파라미터를 지원하지 않고 조용히
  무시**한다(공식 문서에도 파라미터로 정의돼 있지 않음 — 에러도 안 나서 눈치채기 어려움).
  그래서 offset을 아무리 밀어도 "현재 유효 주문 목록" 범위(약 13~19건, 최근 2주)를 못 벗어났고,
  이걸 "리텐션이 2주"라고 잘못 결론 내렸었음. 실제로는 **`orders_timerange` /
  `my_trades_timerange`라는 별도 전용 엔드포인트**가 있고, 여기서 `from`/`to`가 진짜로
  동작하며 기본 **6개월(약 180일)**까지 지원한다(공식 SDK 문서 확인: "Historical orders, by
  default, only data within the past 6 months is supported". `account_book`도 독립적으로
  180일 한도 — 사용자가 웹 UI에서 본 "past 180 days" 문구와 일치). 실측으로 179일 윈도우를
  줘서 페이징해보니 실제로 2026-02-21까지(약 4.5개월 전) 진짜 과거 데이터가 나왔고,
  `account_book`에서 찾은 최초 거래 시점과 정확히 일치함 → 이게 이 계정의 진짜 최초 시점(180일
  한도 안에서는)으로 확인됨.
  **교훈: 특정 파라미터를 넘겼는데 에러 없이 무시되는 경우가 있다 — 응답의 실제 값(타임스탬프
  등)이 요청한 조건을 만족하는지 반드시 확인할 것.** 정상 200 응답이 왔다고 그 파라미터가
  실제로 반영됐다는 보장이 없음. `src/exchanges/gateio/endpoints.js`/`fetcher.js`는
  `orders_timerange`/`my_trades_timerange` + 179일 윈도우로 수정 완료.
- **증분 동기화(sync_state) 생략 결정**: 계정 1~2개 저빈도 개인계정 규모라 매 refresh마다
  179일 전체 윈도우를 재조회 후 upsert(INSERT OR REPLACE)하는 것으로 충분하다고 판단해
  deepcoin의 `sync_state`/`firstSync`/`begin=` 증분 커서 로직은 이식하지 않음. 180일보다 오래된
  과거 데이터는 애초에 API로 가져올 방법이 없으므로, 지금부터 매일 refresh를 계속 돌리는 게
  180일 넘는 히스토리를 잃지 않는 유일한 방법(deepcoin과 동일한 원칙, §3 참고). 계정 수나 거래
  빈도가 크게 늘면 재검토 필요(§7 참고).
- **레버리지는 애초에 안 씀 (사용자 결정, 2026-07-10)**: deepcoin은 레버리지별 통계 분류가

- **리베이트/커미션**: Gate.io 리베이트 정산 규칙은 API로 안 나옴 — 계정별 실측 결과(2026-07-10):

  한정 웰컴 바우처가 이미 소진된 것**으로 결론. 수수료율 자체(mkfr 0.02%/tkfr 0.05%)도 할인
  없이 표준가 그대로라 진행형 혜택도 없음. `main`은 애초에 레퍼럴 관계 자체가 없음
  (`rebate/user/info`의 `invite_uid: 0`). 그래서 `rebate=0` 유지, 대신 아래 §13 기능으로
  사용자가 직접 추정 요율을 입력할 수 있게 함.
- **계정 관리는 .env가 아니라 대시보드 설정 모달로 (사용자 결정, 2026-07-10)**: API 키를

  등록/삭제하도록 재설계함. 키 등록 시 저장 전에 실제로 거래소에 인증 호출을 한 번 해서 유효성과

  `orders_history`/`positions`의 과거 데이터는 그대로 남고, 활성 계정 목록(`credentials` 테이블)
  에서만 빠지므로 화면에 안 보임 → 나중에 같은 키를 다시 등록하면 refresh 없이도 바로 복귀됨
  (실측 확인됨). `.env`의 `GATEIO_API_KEY`는 이제 "최초 1회 마이그레이션용"으로만 쓰이고 서버
  시작 시 credentials 테이블이 비어있으면 자동으로 옮겨간 뒤 더 이상 참조 안 함.

## 13. Deepcoin 어댑터 추가 + 계정 식별 복합키 수정 + 리베이트 요율 수동 입력 (2026-07-10)

- **Deepcoin도 같은 설정 모달로 등록 가능**: `src/exchanges/deepcoin/*` 어댑터 추가,
  `src/exchanges/index.js`에 거래소 레지스트리를 둬서 서버가 `exchange` 필드로 어댑터를 디스패치.
  스키마 자체를 원래 deepcoin 기준으로 설계해둔 덕에(§ARCHITECTURE.md) 필드 매핑이 거의 그대로
  됨 — Gate.io처럼 체결(fills)을 따로 합산할 필요 없이 orders-history에 fee가 이미 있음.
  실제 `../deepcoin/.env`의 진짜 키로 등록→refresh→82주문/25트레이드 수집까지 검증 완료(테스트
  후 삭제, DB엔 흔적 없음). Deepcoin은 passphrase가 추가로 필요해서 `credentials` 테이블에
  `passphrase` 컬럼 추가(기존 DB는 서버 시작 시 자동 ALTER TABLE로 마이그레이션).
- **버그 사전 발견: 계정 필터가 라벨만 보고 있었음.** 거래소가 여러 개면 같은 라벨(둘 다 "main"

  `(exchange, label)` 복합키(`"exchange::label"`) 기준으로 전부 고침. 대시보드 토글의 체크박스
  value도 이 복합키를 씀.
- **리베이트 요율 수동 입력 기능**: 거래소가 API로 리베이트/바우처 비율을 안 알려주므로, 설정
  모달에서 계정별로 `credentials.rebateRate`(0~1)를 직접 입력하게 함. **`orders_history`의 실제

## 14. Deepcoin 페이지네이션 버그 + "10일치만 나옴" 문제의 진짜 원인

**[전면 정정 — 2026-07-11, `../deepcoin` 세션에서 재조사 후 확정]**
아래 2026-07-10 시점의 결론(짧은 페이지 가정 수정 + "이건 버그가 아니라 진짜 리텐션 한계"
재검증)은 **최종 결론이 틀렸었다.** 두 가지 다 같은 근본 원인 하나를 놓친 결과였음:

> **`before` 커서 파라미터가 `orders-history`/`trade/fills` 엔드포인트에서 사실상 무시된다.**
> 어떤 값을 `before=`로 보내도 매번 "최근 limit개"만 돌려준다. 그래서:
> - 페이지를 아무리 넘겨도 새 데이터가 아니라 **직전 응답과 대부분 겹치는 데이터**만 돌아옴
>   (겹치는 부분이 조금씩 빠지면서 응답 크기가 100→99→98...로 줄어드는 것처럼 "보였을" 뿐,
>   실제로는 새로운 과거 데이터가 아니라 그냥 덜 겹치는 중복이었음)
> - 그래서 아무리 재시도/재검증해도 항상 "최근 며칠"에서 못 벗어났던 것 — API가 진짜 그것밖에
>   없어서가 아니라 **커서가 매번 리셋되고 있었기 때문**
>
> 확정 실측(`limit`을 20/50/100으로 바꿔가며 `before`로 여러 페이지 수집 → 매번
> **distinct 개수가 정확히 limit값과 일치** = 새 데이터가 단 한 건도 없었다는 수학적 증거).
> `after`로 바꾸자 즉시 정상화: 6개월치(723건, 중복 0, 마지막엔 진짜 빈 페이지로 자연 종료)가
> 한 번에 다 잡힘. `begin`/`end` 파라미터를 62~90일 전으로 줘도 무시된다고 결론 내렸던 것도,
> 애초에 페이지네이션 자체가 씹히고 있었으니 당연히 아무 의미 없는 테스트였음.

**수정**: `src/exchanges/deepcoin/endpoints.js`의 `ordersHistory.pagination.cursorParam`을
`'before'` → `'after'`로 변경. `client.js`의 "짧은 페이지도 끝이 아닐 수 있다"는 관용적 종료
조건은 해로울 게 없어 그대로 뒀지만(가끔 요청 하나 더 나가는 정도), **그게 진짜 원인이라고
오해하지 말 것** — 실제 원인은 커서 파라미터 이름이었다.

**교훈 (여전히 유효함, 결론만 바뀜)**: 사용자가 "예전엔 달랐다"고 구체적으로 반박하면 다시
실측할 것 — 다만 이번엔 재검증 방법 자체(같은 `before` 커서로 더 세게/더 오래 시도)가 틀려서
같은 함정에 두 번 빠졌다. **재검증할 땐 가정을 하나 더 의심할 것**: "더 세게 시도했는데도 안
된다"는 "한계에 도달했다"가 아니라 "애초에 이 방법 자체가 안 먹힌다"일 수도 있다. 이번 경우
distinct-count 비교(`limit`을 바꿔가며 결과 개수가 `limit`과 정확히 일치하는지 확인)처럼
"진짜 새 데이터가 오고 있는가"를 직접 검증하는 게 훨씬 빠른 판별법이었음.

**결과적으로 "과거 깊은 히스토리가 필요하면 `../deepcoin`의 기존 DB를 마이그레이션해야 한다"는
아래 결론도 불필요해짐** — 이제 그냥 refresh 한 번이면 Deepcoin API에서 직접 6개월치를
받아온다.

## 15. `state='filled'` 필터가 부분체결-후-취소 주문을 놓친다 (§6 트레이드 그룹핑 버그의 진짜 원인, 2026-07-11)

§14로 페이지네이션은 고쳤는데도 §6에서 겪었던 "포지션이 안 닫히는" 뭉침 버그가 그대로 남아있어서
(`after`로 6개월치를 다시 받아도 그 지점만 여전히 안 맞음) `../deepcoin` 세션에서 재조사함.

**진짜 원인**: 그 지점에 지정가 주문이 **1계약만 부분체결되고 나머지는 취소**됐었음
(`state: "partially_filled_canceled"`, `accFillSz: 1`, 실현손익 있는 진짜 거래). 근데 모든
쿼리가 `WHERE state = 'filled'`로 필터링하고 있어서, 상태 문자열이 정확히 `'filled'`가 아닌
이 주문은 DB엔 멀쩡히 있는데 통계에서만 조용히 빠지고 있었음. 전체 664건 중 딱 1건뿐이었지만
그게 이후 3주치 트레이드를 통째로 하나로 뭉개버렸음(FIFO 사이즈 장부가 영원히 안 맞게 됨).

**수정 (이미 적용됨)**: `src/server.js`의 `WHERE state = 'filled'` 4곳 전부
`WHERE CAST(accFillSz AS REAL) > 0`으로 교체 — "최종 상태 문자열"이 아니라 "실제 체결
수량"이 통계 포함 여부의 진짜 기준.

**✅ Gate.io 검증 완료 (2026-07-11, trade-health-tracker에서 실측)**:
1. `o.size`는 확인 결과 "주문 요청 수량"이었고, 실제 체결량은 별도 필드 `o.left`(잔량)와의
   차이(`abs(size) - abs(left)`)로 계산해야 정확함 — 우려했던 대로 반대 방향 오류
   (미체결분까지 실현손익에 낀 것처럼 취급) 소지가 실제로 있었음. `exchanges/gateio/fetcher.js`의
   `normalizeOrders`에서 `accFillSz: Math.abs(size)` → `Math.abs(size) - Math.abs(o.left || 0)`로 수정.
2. 실측 계정(`chacha`)에서 `finish_as` 값 분포 확인: `filled`(363) / `reduce_out`(65) /
   `cancelled`(51). 후자 116건은 전부 `size === left`(체결량 0인 완전 미체결)이라 이번 계정
   기준으론 §11류 버그가 실제로 발생하진 않았음 — 다만 §15의 필터 수정(`accFillSz>0`)과 이번
   `accFillSz` 계산 수정 둘 다 앞으로 부분체결 주문이 생겨도 안전하도록 미리 고쳐둔 것.

**교훈**: 거래소 주문 `state`/`finish_as` enum은 성공/실패 이분법이 아니라 훨씬 세분화돼 있을
수 있다. **상태 문자열로 필터링하지 말고 실제 체결 수량으로 필터링할 것.** 발생 빈도가 극히
낮아서(664건 중 1건) 합계 검증으론 거의 안 걸리고, "특정 포지션이 왜 안 닫히지?" 같은 구조적
이상 증상으로만 드러남 — 승률/평균손익 등 트레이드 단위 지표를 만들 계획이면 처음부터
`accFillSz` 필드의 정확성을 의심하고 실측해둘 것.
